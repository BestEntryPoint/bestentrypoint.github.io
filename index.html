Pocket Commanders
 As we already know, in programming we can control virtual objects.

Reality is stunning. In programming, we can control virtual objects in very diverse ways.

Just like in your regular life you can control your candy:
· can put the candy in the pocket
· can be storing the candy inside the pocket
· can take the candy from the pocket



in programming, while running code, we can also control virtual objects the same way.

So, for example, in the same way we can control the virtual object 5:
· can put the object 5 in the pocket
· can be storing the object 5 inside the pocket
· can take the object 5 from the pocket



In the same way, we can control the virtual object Hello, for example:


Of course, such pockets, in programming, are virtual and exist only in the mind of the machine. We, of course, cannot feel them by our hands. But they really exist and can store virtual objects.

For example, to control the virtual object 5 in the way described above, we first must create a special Commander -- Pocket Commander -- which, in turn, makes the machine create a virtual pocket for storing that virtual object.

Pocket Commanders are called this way not because they are miniature or capable of being carried in one's pocket, but only because they make the machine create virtual pockets for storing virtual objects.

Interestingly, when creating any Commanders (of course, incl. Pocket Commanders) we must give them names just like parents give names to their kids when the kids are born.

There are 2 sorts of Pocket Commanders that we might create:
· Invariable-Pocket Commanders
· Variable-Pocket Commanders.

Invariable-Pocket Commanders
 If we create an Invariable-Pocket Commander it means that then we cannot change the object controlled by this Commander. This will be more clear with examples below.

So, for example, to create our own Invariable-Pocket Commander named Sam we first must write val and let, respectively, before the Commander’s name:

  fun main() {
    val Sam  

}
let Sam  

 

Next, to put the object 5 in control of the Invariable-Pocket-Commander Sam we must further write the equal sign = and then 5:

  fun main() {
    val Sam = 5
    
}
let Sam = 5

 

Don't let the equal sign = confuse you. This single line of code:
      val Sam = 5
let Sam = 5  

only means that the Invariable-Pocket Commander Sam takes the object 5 in its control.

Specifically, that single line of code above means that the Invariable-Pocket Commander Sam makes the machine do the 3 things there:
· create the invariable pocket Sam
· put the object 5 in the invariable pocket Sam
· start storing the object 5 inside the invariable pocket Sam

Please distinguish between the Invariable-Pocket Commander Sam and the invariable pocket Sam:


For example, our whole code might look like this:

  fun main() {
    val Sam = 5
    print(Sam)
}
let Sam = 5
print(Sam)
 
         
  5   5

Don't let the equal sign = confuse you again. This single line of code:

var Sam = 5

Specifically, that single line of code above means that the Variable-Pocket Commander Sam makes the machine do the 3 things there:

Here you might ask me: Why ever should we create Invariable-Pocket Commanders, since Variable-Pocket Commanders can do everything they do?

The answer is that there is one notable thing about the programming process. The programmers (coders) spend a lot of their time reading code.

So while coding you will be spending a lot of your time reading your own or someone else'code and in the case there is a lot of Variable-Pocket Commanders it will be very hard for you to understand how exactly the program works, even if it is your own program written just a little while ago. It is so since you will need to check each Variable-Pocket Commander at each point of the program for changes to its object.

So from this perspective, the most experienced programmers recommend creating Invariable-Pocket Commanders (val and let, respectively) always when it is just possible.
