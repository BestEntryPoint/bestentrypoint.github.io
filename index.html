Commanders
 As we already know, programming is a wonderland where we can make machines do the works we want by just writing some texts named codes.


But how, exactly, should codes be arranged inside? What is THE principle there?

Right now, we will get that.

In programming, we command the machine only by Commanders:


So it is namely Commanders that make the machine do the works that we want. They are our helpers.

Interestingly, Commanders (our helpers) exist only in the mind of the machine:


Such commanders are called virtual commanders. We cannot feel them by our hands. But they really exist and can make the machine do the works we want.

So, thankfully, we can explicitly interact with such virtual commanders.

Each Commander has a unique name:


and there is a lot of Commanders with different names.

Importantly, each Commander can make the machine do some specific work.

Just like on the street, when we need specific persons we might call (invoke) them by their names:


in programming, when we need specific Commanders to make the machine do specific works we also can call (invoke) them by their names.

Of course, in programming, we can call (invoke) Commanders only by writing their names in code:

Pocket Commanders
 Just like we can control a candy:
· can put the candy in the pocket
· can be storing the candy inside the pocket
· can take the candy from the pocket



in programming, while running code, we can control virtual objects the same way.

So, for example, we can control the virtual object 5:
· can put the object 5 in the pocket
· can be storing the object 5 inside the pocket
· can take the object 5 from the pocket



Also, for example, we might control the virtual object Hello the same way:


Of course, such pockets, in programming, are virtual and also exist only in the mind of the machine. We also cannot feel them by our hands. But they really exist and can store virtual objects.

For example, to control the object 5, we must create a special Commander -- Pocket Commander -- which automatically makes the machine create a virtual pocket for that virtual object.

Pocket Commanders are called this way not because they are miniature or capable of being carried in one's pocket, but only because they automatically make the machine create virtual pockets for storing objects.

Interestingly, when creating any Commanders (of course, incl. Pocket Commanders) we must give them names just like parents give names to their kids when the kids are born.

Pocket Commanders are used to associate a name (Sam or John) with a value (e.g. the number 5 or the string "Hello").

There are 2 sorts of Pocket Commanders that we might create:
· invariable Pocket Commanders
· variable Pocket Commanders.

Invariable Pocket Commanders
 If we create an invariable Pocket Commander it means that then we cannot change the object controlled by this Commander. This will be more clear with examples.

So, for example, to create our own invariable Pocket Commander named Sam we must write val or let before the Commander’s name:

  fun main() {
    val Sam  

}
let Sam  

 

Next, to put the object 5 in control of the invariable Pocket-Commander Sam we must further write the equal sign = and then 5:

  fun main() {
    val Sam = 5
    
}
let Sam = 5

 

Don't let the equal sign = confuse you. This single line of code:
      val Sam = 5
let Sam = 5  

only means that the invariable Pocket Commander Sam takes the object 5 in its control.

Specifically, that single line of code above means that the invariable Pocket Commander Sam makes the machine do the 3 things there:
· create the invariable pocket Sam
· put the object 5 in the invariable pocket Sam
· start storing the object 5 inside the invariable pocket Sam

Please distinguish there the invariable Pocket Commander Sam and the invariable pocket Sam:


For example, our whole code might look like this:

  fun main() {
    val Sam = 5
    print(Sam)
}
let Sam = 5
print(Sam)
 
         
  5   5
